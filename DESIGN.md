# MarketCrew Reboot Design

## 목표

기존 MVP 구현을 지우고, 연결이 살아 있는 빈 운영실에서 다시 시작한다.

## 첫 기준선

- 대표 로그인
- 운영 상태 화면
- API health
- Vercel에서 Railway API로 이어지는 bridge
- Railway Postgres 연결 확인
- provider env 존재 여부 확인

## 현재 구현 진행

- `search-ad-operations-foundation` module-1~8 Do 로컬 구현 완료.
- 왼쪽 메뉴 기반 SaaS 셸과 검색광고 운영 홈을 만들었다.
- 브랜드는 `전체`, `커피프린트`, `스티커씨`로 필터링하되, `전체`는 비교가 아니라 운영 현황 합산으로만 쓴다.
- 광고유형은 `파워링크`, `쇼핑검색광고`로 분리한다.
- 네이버 보고서는 원문 파일, 원문 행, 정규화 행을 분리 저장하도록 schema를 잡았다.
- 보고서 보관함과 상세 화면에서 요약, 쉬운 표, 문제 후보, 좋은 후보, 원본 미리보기, 컬럼 설명을 보여준다.
- Search Ad 클라이언트는 읽기와 보고서 다운로드까지만 구현했다. 실제 provider write는 아직 열지 않는다.
- 캠페인/광고그룹/키워드 상태 스냅샷 화면을 추가했다.
- 검색어 성과는 정규화 보고서 행 기준으로 비용, 클릭, 전환, CPA, ROAS를 표시한다.
- 규칙 엔진은 최소 표본 기준으로 저효율, 클릭 없음, 높은 CPA, 낮은 ROAS, 우수 후보를 만든다.
- ON/OFF 요청은 실행 미리보기와 실행 이력으로 남기며, write gate가 닫혀 있으면 네이버에는 반영하지 않는다.

## 제외

- 기존 캐릭터 데스크
- 기존 결재/실행/성과 카드
- 기존 workflow 데이터
- 외부 provider write

## 다음 설계 후보

1. 네이버 검색광고 보고서 자동 수집: `/stat-reports` 작업 생성, 상태 확인, 인증 헤더 포함 다운로드, 원본 보관
2. 보고서 보관함과 상세 화면: 어려운 보고서를 한국어 요약, 쉬운 표, 컬럼 설명으로 확인
3. 보고서 원본 보관: 1차 MVP는 Postgres에 원본 텍스트와 원본 행을 저장하고, 데이터가 커지면 외부 스토리지로 분리
4. 보고서 파서: 헤더 없는 탭 구분 파일을 `reportTp`별 컬럼 순서표로 해석
5. 브랜드/채널 데이터 수집 구조 재정의
6. 검색광고 키워드 규칙 엔진 재설계
7. 성과 기준 확인 화면: 기간, 최소 표본, CPA/ROAS, 실행 단계 기준을 대표가 확인
8. LLM 판단 근거 패킷 재설계
9. 대표 승인과 실행 게이트 재설계
