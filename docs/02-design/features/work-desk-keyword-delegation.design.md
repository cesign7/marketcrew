# 업무카드형 데스크와 키워드 위임 디자인

## 목적

검색광고 조정 안건을 긴 문서 하나로 보지 않고, 실제 대표가 판단해야 하는 키워드/검색어 단위의 업무카드로 쪼갠다. 각 카드는 담당 캐릭터, 성과 근거, 제안 조치, 유지 예외, 첫 승인 여부를 한눈에 보여준다.

## 정보 구조

- 왼쪽 메뉴에는 `직원 데스크`를 별도 구역으로 두고 모아, 그로, 마루, 프로, 리피, 카피, 데이로 바로 이동한다.
- `/operations`는 오늘 볼 카드만 최대 6개로 요약한다.
- `/characters/gro`는 검색광고와 쇼핑검색광고 키워드 조정 카드를 보여준다.
- `/characters/day`는 전환 추적, 주문 연결, 근거 누락처럼 조정 전에 확인해야 하는 카드를 보여준다.
- `/characters/moa`는 대표 첫 승인이 필요한 카드와 향후 위임 후보를 모아 보여준다.

## 카드 계약

`WorkDeskCardView`는 화면용 계약이다.

- `keywordLabel`: 실제 키워드 또는 쇼핑검색어
- `brandLabel`: 스티커씨 또는 커피프린트
- `domainLabel`: 검색광고 또는 쇼핑검색광고
- `ownerId`, `ownerName`: 담당 캐릭터
- `metricLabels`: 클릭, 비용, 주문, CPA, 직접 전환율 같은 핵심 숫자
- `contextLabels`: 캠페인, 광고그룹, 기기, 시간대, 상품그룹
- `diagnosisLabel`: 규칙 엔진이 잡은 문제 유형
- `recommendedActionLabel`: 입찰 하향, 일시중지 검토, 기기 조정, 시간 조정, 추적 확인 등
- `reasonLabel`: 저성과라도 유지할 수 있는 예외나 조정 이유
- `delegation`: 대표 첫 승인, 데이터 확인, 모아 위임 후보 상태
- `evidenceLabels`: 연결 근거 ID 목록

## 데이터 흐름

```text
ProviderSyncReport
  -> SearchAdPerformanceSnapshot / ShoppingSearchAdPerformanceSnapshot
  -> buildAdPerformanceDiagnoses
  -> WorkDeskCardView
  -> /operations 요약 카드
  -> /characters/{id} 담당자별 데스크
```

규칙 엔진은 LLM보다 먼저 확정 가능한 성과 문제를 잡는다. LLM은 카드의 사유 보강, 유지 예외, 대표 보고 문구를 풍부하게 만드는 방향으로 확장할 수 있지만, 자동 집행 여부는 카드의 구조화된 위임 상태와 write gate가 결정한다.

## 위임 원칙

- 처음 실행하는 조정 유형은 항상 대표 첫 승인이 필요하다.
- 대표가 승인하면서 같은 브랜드, 같은 진단, 같은 조정 한도는 다음부터 모아에게 위임할 수 있다.
- 위임 후에도 모아는 처리 결과를 대표에게 계속 보고한다.
- 전환 추적이 불확실한 카드는 데이가 확인하기 전까지 자동 조정하지 않는다.
- 이번 범위에는 실제 provider write executor를 넣지 않는다.

## 화면 원칙

- 카드 제목에는 키워드가 먼저 보인다.
- 본문은 2~3줄 안에서 진단과 조치를 읽게 한다.
- 긴 캠페인/광고그룹/상품명은 줄바꿈과 말줄임으로 화면을 깨지 않게 한다.
- 모바일에서는 1열 카드로 접고, 왼쪽 메뉴의 직원 데스크는 가로 스크롤 가능한 간결 링크로 유지한다.
- 영어 내부 용어는 화면에 직접 노출하지 않는다.

## 검증

- 키워드별 카드 생성 단위 테스트.
- 업무카드 컴포넌트 렌더 테스트.
- 왼쪽 메뉴와 직원 데스크 이동 Playwright 스모크.
- 전체 타입 검사, 단위 테스트, 빌드, 감사 검사.
